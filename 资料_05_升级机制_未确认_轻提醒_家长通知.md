# 智能学习系统资料 5：升级机制 — 未确认 → 轻提醒 → 家长通知

## 来源
- 标题：用 OpenClaw 给孩子搭一套全科辅导系统（Plus版⑤）
- 作者：苹果落地
- 来源：人间折腾录
- 日期：2026 年 4 月 16 日

---

## 一、本篇目标

解决最现实的问题：**开始了，但没人确认。**

**验收点**：

| 场景 | 预期 |
|------|------|
| 超过设定时间未确认"已开始" | 触发轻提醒 |
| 再超阈值仍未确认 | 通知家长（仅摘要） |
| 22:00 后 | 默认静默，不播报音响，只保留灯光提示 + 家长通知 |

> 目标不是"怎么提醒更多次"，而是分阶段升级 + 合适时停下来。

---

## 二、为什么必须单独做

执行层解决"设备会不会动"，不解决"人会不会回应"。

**真实场景常见问题**：
- 开始了，但孩子没回"已开始"
- 提醒了一次，还是没人回应
- 到了夜间，不适合继续播报
- 家长需要知道发生了什么，但不希望群里刷屏

> 不先定好确认/升级/夜间策略，系统会变成两种极端：一直提醒停不下来，或完全不提醒等于没开始。

---

## 三、技术栈与依赖

| 组件 | 职责 |
|------|------|
| `skills/study-ack` | 接收"已开始"/"已完成" |
| `skills/study-escalation` | 定时检查并决定是否升级 |
| `data/routines/policies.json` | 策略配置（阈值参数化） |
| `data/routines/state.json` | 状态文件（扩展字段） |
| `data/family/contacts.json` | 家长通知对象（可选） |

**依赖关系**：
- state.json → 记录是否在学习、是否已确认、当前升级阶段
- policies.json → 控制多久提醒、多久通知家长、夜间是否静默

---

## 四、策略文件 data/routines/policies.json

**必须参数化，阈值不写死在 Skill 里。**

```json
{
  "pomodoro_minutes": 25,
  "break_minutes": 5,
  "ack": {
    "start_required": true,
    "complete_required": true
  },
  "escalation": {
    "no_ack_minutes": 5,
    "second_notice_minutes": 10,
    "max_stage": 2
  },
  "night": {
    "enabled": true,
    "start": "22:00",
    "end": "06:00",
    "silent_speaker": true,
    "light_only": true
  },
  "reminder_intensity": "low"
}
```

**关键字段**：
- `no_ack_minutes`：开始后多久没确认触发轻提醒
- `second_notice_minutes`：再过多久通知家长
- `night.silent_speaker`：夜间不播报音响
- `night.light_only`：夜间只做灯光提示
- `reminder_intensity`：映射不同提醒脚本

> 后面做"组合计划"时，实际上就是在改这个文件。

## 六、HA 两个提醒脚本

### 1️⃣ script.study_soft_nudge — 轻提醒

```yaml
script:
  study_soft_nudge:
    alias: "Study Soft Nudge"
    mode: restart
    sequence:
      - service: light.turn_on
        target:
          entity_id: light.study_lamp
        data:
          brightness_pct: 70
      - service: tts.google_translate_say
        target:
          entity_id: media_player.xiaoai_speaker
        data:
          message: "提醒一下：现在开始第一关。开始后在群里回复：已开始。"
```

> 不要写成长播报，不要做强刺激。目标是"提醒一次"，不是"强行施压"。

---

### 2️⃣ script.study_night_silent_nudge — 夜间静默

```yaml
script:
  study_night_silent_nudge:
    alias: "Study Night Silent Nudge"
    mode: restart
    sequence:
      - service: light.turn_on
        target:
          entity_id: light.study_lamp
        data:
          brightness_pct: 50
          color_temp: 400
      - delay: "00:00:01"
      - service: light.turn_off
        target:
          entity_id: light.study_lamp
      - delay: "00:00:01"
      - service: light.turn_on
        target:
          entity_id: light.study_lamp
        data:
          brightness_pct: 50
          color_temp: 400
```

> 夜间依然保留反馈，但不制造噪音。

## 七、study-ack：确认机制

**路径**：`skills/study-ack/SKILL.md`

```markdown
## 指令
- /确认 已开始
- /确认 已完成
- 也支持自然语言：
  - "已开始" / "我开始了"
  - "已完成" / "做完了"

## 行为
1) 读取 sender_openid → member_id
2) 读取 state.json，确认 mode 是否为 study
3) 若"已开始"：
   - state.session.ack_start = true
   - state.session.ack_start_at = now
4) 若"已完成"：
   - state.session.ack_done = true
   - state.session.ack_done_at = now
5) 停止升级：
   - state.session.escalation_stage = 0
   - state.session.last_nudge_at = null

## 输出
- kid：短输出 ≤6 行："✅ 已确认，继续第1关。"
- parent/admin：看到摘要
```

**核心原则**：一旦确认，必须让升级链路停下来。否则：孩子已确认，系统还在继续提醒。

## 八、study-escalation：升级机制（周期检查）

**路径**：`skills/study-escalation/SKILL.md`

> **靠定时检查，不靠聊天触发。**

```markdown
## 指令
- /升级检查（管理员调试用）
- （推荐）定时任务：每 1 分钟运行一次

## 核心逻辑（必须）
1) 读取 state.json：
   - 若 mode != "study"：退出
   - 若 session.ack_start == true：退出（不再升级）

2) 读取 policies.json：
   - no_ack_minutes / second_notice_minutes / night 策略

3) 计算 elapsed_min = 当前时间 - session.started_at

4) 判断夜间：
   - 若 night.enabled 且时间在 [night.start, night.end)：
     - 使用夜间脚本（不播报音响）
     - 只做灯光提示 + 家长通知

5) 分阶段执行：
   - 若 elapsed_min >= no_ack_minutes 且 stage < 1：
     - 触发 HA：script.study_soft_nudge（或夜间脚本）
     - escalation_stage = 1
     - 记录 last_nudge_at
   - 若 elapsed_min >= second_notice_minutes 且 stage < 2：
     - 通知家长（摘要）
     - escalation_stage = 2
     - 记录 last_notify_at

6) 所有阶段写回 state.json（必须落盘）

## 家长通知内容（必须短）
- 谁：current_member_id
- 状态：已开始未确认（已过去 X 分钟）
- 建议：请在群里提醒孩子回复"/确认 已开始"

## HA 调用
- POST /api/services/script/turn_on
- body: {"entity_id": "script.study_soft_nudge"}
  或 {"entity_id": "script.study_night_silent_nudge"}
```

**关键判断顺序（不能写反）**：

```
1. 先看 mode 是不是 study
2. 再看是否已经 ack_start=true
3. 然后算是否到阈值
4. 最后判断当前第几阶段
```

---

## 九、定时检查必须跑起来

> **最稳做法：每 1 分钟执行一次 study-escalation**

如果还没接好定时任务，先走调试模式：管理员手动发 `/升级检查`

> 先验证逻辑，再接自动化，排错成本最低。**这一步不要跳过。**

---

## 十、状态流转

```
开始学习
    → mode=study, started_at 写入, ack_start=false, escalation_stage=0

超过 no_ack_minutes 未确认
    → 触发轻提醒，escalation_stage=1

超过 second_notice_minutes 仍未确认
    → 通知家长，escalation_stage=2

孩子确认"已开始"
    → ack_start=true, escalation_stage=0（降级）

孩子确认"已完成"
    → ack_done=true, escalation_stage=0（降级）
```

---

## 十一、验收步骤

### Step 1：启动学习但不确认
```
群发：/开始学习
```
**检查**：`mode=study`, `started_at` 已写入, `ack_start=false`

### Step 2：触发第一次升级
```
管理员发：/升级检查
```
**检查**（已超过 no_ack_minutes）：
- 白天：触发 `script.study_soft_nudge`
- 夜间：触发 `script.study_night_silent_nudge`
- `escalation_stage=1`

### Step 3：触发第二次升级
```
继续不确认，再执行：/升级检查
```
**检查**（已超过 second_notice_minutes）：
- 家长收到摘要通知
- `escalation_stage=2`

### Step 4：孩子确认
```
群发：/确认 已开始
```
**检查**：
- `ack_start=true`
- `escalation_stage=0`
- 后续 `/升级检查` 不再触发提醒

### Step 5：测试夜间策略
**检查**：
- 不播报音响
- 仅灯光提示
- 家长通知仍可保留

---

## 十二、常见问题

| 问题 | 排查 |
|------|------|
| 没确认但系统不提醒 | 检查：started_at 是否写入 / mode 是否为 study / study-escalation 是否被执行 |
| 夜间还是会播报音响 | 检查：night.silent_speaker 是否为 true / 时间是否真的在夜间区间 / 跨天逻辑 |
| 提醒停不下来 | study-ack 没写回 ack_start=true / study-escalation 没先判断 ack_start |
| 家长通知怎么发 | 暂时支持不了点对点 → 用群内 @ 家长，先跑通"通知发生了" |

> 夜间判断最容易写错的就是**跨天逻辑**（22:00-06:00）。

---

## 十三、小结

**本篇完成后，学习系统才真正具备了"可靠性"。**

**补齐三层能力**：

| 层级 | 能力 |
|------|------|
| 执行层 | 开始、节奏、结束 |
| 确认层 | 已开始、已完成 |
| 升级层 | 未确认 → 轻提醒 → 家长通知 |

**再加夜间静默策略**，系统就不会因为"自动化"而变成持续打扰。

**核心价值**：不是多加一个提醒脚本，而是让整套执行系统具备了**可依赖的推进能力**。

# 智能学习系统资料 3：学习主干 — 作业拆解 + 双视图 + 落盘

## 来源
- 标题：用 OpenClaw 给孩子搭一套全科辅导系统（Plus版③）
- 作者：苹果落地
- 来源：人间折腾录
- 日期：2026 年 4 月 15 日

---

## 一、本篇目标（验收标准）

完成后必须满足以下三点：

1. ✅ 群里粘贴作业后，系统拆成 **5-20 分钟**颗粒任务
2. ✅ 输出双视图：
   - **孩子版**：短句、可执行、可勾选
   - **家长版**：顺序、估时、完成标准、检查点 ≤ 3 个
3. ✅ 自动落盘两份文件：
   - `data/homework/inbox/` — 原始输入
   - `data/homework/parsed/` — 结构化计划

> 缺一个，都不能算闭环完成。

---

## 二、为什么这一层要先做

执行层依赖的不是"作业原文"，而是"结构化任务"。

| 下游依赖 | 需要的输入 |
|----------|-----------|
| 音响播报 | 当前是哪一关 |
| Timer 推进 | 单个任务大概多久 |
| 日报生成 | 今日原始输入和拆解结果 |
| 周报统计 | 任务类型、时长、完成情况 |

**本质**：建立学习主干的数据入口。

---

## 三、前置条件

- ✅ 已完成 openid → member_id → role 的身份路由（资料 2）
- ✅ OpenClaw 具备 workspace 写文件权限

> 落盘时必须带 member_id，输出时必须按 role 控制内容长度。

---

## 四、目录结构

```
data/homework/
├── inbox/          # 原始作业输入
├── parsed/         # 拆解后的计划
└── done/           # 完成归档（预留）
```

> 后续日报、周报、趋势、执行层都会依赖这部分目录，不建议临时命名。

---

## 五、文件命名规范

**固定格式**：`YYYY-MM-DD_<member_id>_<scope>.md`

**示例**：
- `2026-04-14_kid_1_homework.md`
- `2026-04-14_kid_1_plan.md`
- `2026-04-14_kid_1_数学_plan.md`（细分科目）

> 文件名必须包含日期 + member_id，否则多孩子场景下检索和统计会出问题。

---

## 六、homework-intake Skill 最小能力

稳定完成四件事：
1. 接收作业输入
2. 拆成 5-20 分钟颗粒任务
3. 输出孩子版和家长版
4. 将原文和计划分别落盘

---

## 七、输入协议（收敛两种）

### 1️⃣ 常规输入
```
/作业
语文：...
数学：...
英语：...
可用时间：60分钟
状态：一般
```

### 2️⃣ 极简输入
```
/作业 数学口算40题，应用题3题，英语背单词10个
```

**至少提取字段**：
- 科目
- 数量
- 优先级
- 可用时间
- 当前状态（如果有）

---

## 八、拆解规则（必须写死）

| 约束 | 规则 |
|------|------|
| 单任务时长 | **5-20 分钟**（硬约束） |
| 超出 20 分钟 | 继续拆 |
| 小于 5 分钟 | 合并 |
| 默认排序 | **先易后难** 或 **先交付后订正** |

> 任务粒度不稳定，执行层和 timer 就接不住。

---

## 九、双视图（固定模板，不要漂）

### 🧒 孩子版（6-10 行）

只保留执行信息，只解决"下一步做什么"。

**要求**：
- 顺序明确
- 每关一句话
- 有完成标准
- 中间有休息点
- 最后有完成打卡提示

**推荐结构**：
```
第1关：数学口算 40题（15分钟）
第2关：数学应用题 3题（10分钟）
休息：5分钟
第3关：英语背单词 10个（15分钟）
最后：完成后回复"已完成"
```

---

### 👨‍👩‍👧 家长版

看的是执行计划，不是过程陪盯。

**包含字段**：
- 总时长估算
- 任务顺序
- 每项预计分钟
- 每项完成标准
- **检查点 ≤ 3**

**检查点设计原则**：只看结果，不看过程。

**示例**：
- ✅ 数学是否完成并订正
- ✅ 英语是否完成背诵打卡
- ✅ 全部任务是否收尾

> 家长版目标是方便判断是否需要介入，不是让家长全程跟着。

---

## 十、落盘模板（inbox + parsed 分离）

### 1️⃣ inbox — 原始输入

| 字段 | 内容 |
|------|------|
| 原始作业文本 | 孩子/家长发送的原文 |
| 发送时间 | 时间戳 |
| member_id | 发送者身份 |
| role | 发送者角色 |
| 可用时间 | 如果有 |
| 状态 | 如果有 |

### 2️⃣ parsed — 结构化计划

| 字段 | 内容 |
|------|------|
| 科目 | 作业科目 |
| 总时长估算 | 合计分钟数 |
| 子任务列表 | 拆解后的任务 |
| 每项预计时长 | 单任务分钟数 |
| 休息点 | 休息安排 |
| 检查点 | ≤3 个 |
| 孩子版 | 短格式输出 |
| 家长版 | 完整版输出 |

> 后续执行层、日报、周报都从这里读取，不建议只存自然语言。

---

## 十一、身份路由接入

两个关键落地点：

| 场景 | 规则 |
|------|------|
| 落盘路径 | 按 **member_id** 区分 |
| 输出内容 | 按 **role** 控制 |

**默认规则**：
- **kid**：默认看孩子版
- **parent/admin**：默认看摘要，可展开家长版

**收紧策略**：
- 群里只发摘要 + 孩子版
- 家长版通过 `/展开家长版` 获取

---

## 十二、群内输出策略

**短输出，可展开。详细内容只落盘。**

**默认输出示例**：
```
已生成今日学习计划，预计 55 分钟。
先做第1关：数学口算。
完成后按顺序进入下一关。
家长回复 /展开家长版 查看检查点。
```

> 这条规则可复用到日报、周报和升级提醒。

---

## 十三、验收步骤

### Step 1：kid 发作业
```
群里发送：/作业 ...
```
**预期**：返回孩子版或摘要，给出总时长估算，提示可展开家长版

### Step 2：检查落盘
```
data/homework/inbox/当天_kid_1_homework.md
data/homework/parsed/当天_kid_1_plan.md
```
**预期**：两份文件均存在

### Step 3：parent 展开家长版
```
群里发送：/展开家长版
```
**预期**：返回家长版，不重复刷长内容，不影响孩子版节奏

---

## 十四、常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 任务拆得忽大忽小 | 没有把 5-20 分钟写成硬约束 | 超出就拆，不足就并 |
| 输出格式不稳定 | 没有固定模板 | 把孩子版/家长版模板固定下来 |
| 多孩子串台 | 文件名/路径没带 member_id | 不能省 |
| 群里内容过长 | 完整计划一次性发出 | 坚持"短输出+可展开" |

---

## 十五、小结

**本篇完成后，学习系统主干具备最小闭环**：

```
接作业 → 拆任务 → 输出双视图 → 自动落盘
```

**核心价值**：后面的系统终于有了统一输入。

| 下游模块 | 从 parsed 获取什么 |
|----------|-------------------|
| 执行层 | 当前任务是什么 |
| 设备层 | 下一步播报什么 |
| 报表层 | 今天处理了什么 |
| 趋势层 | 后续怎么统计 |

**前两篇总结**：
- 资料 2：解决了"识别人"
- 资料 3：解决了"识别任务"

> 到这里，学习系统的主干才算真正搭起来。
